# README 生成器

按同目录下的 `README.template.md` 模板，在项目根目录生成或更新 `README.md`。

## 前置

- 模板文件：`./README.template.md`（与本 SKILL.md 同目录）
- 如果模板不存在，告知用户后停止。

## 输出格式

生成的文件直接替换项目根目录的 `README.md`。输出格式要求：
- 所有 `<!-- HTML 注释 -->` 仅模板中有，生成的 README 中**不留模板注释**
- 残留的 `{{PLACEHOLDER}}` 视为 bug，阶段 4 必须逐一清除
- 代码块语言标记正确（bash / json / typescript 等）
- 目录锚点可跳转（中文标题的锚点用 `#标题文本` 格式）

## 执行流程

### 阶段 0：检查现状

- 若根目录已有 `README.md` → 先读取，后续增量更新（已有且正确的内容保留）
- 若没有 → 全量生成

### 阶段 1：信息收集

并行读取以下关键文件（使用 `explore` 子代理或并行 `read_file` 加速）：

1. `package.json`（或等效的项目描述文件）
2. 入口文件（`src/main.ts` / `main.go` / `app.py` / `index.ts` 等）
3. 根模块 / 路由注册 / 中间件注册文件
4. `src/config/`、`.env.example`、环境变量加载代码
5. 通用层目录（`ls` 获取结构：common / shared / core / lib / infra 等）
6. 业务模块目录（递归一级即可）
7. 每个模块的 controller + service + schema（至少读头部注释和路由定义）
8. Dockerfile、CI 配置文件
9. 测试配置（package.json 中的 jest/mocha/vitest 段落）
10. AGENTS.md / CONTRIBUTING.md（提取开发规范和变更流程）

### 阶段 2：系统架构分析

这是 README 的灵魂。按模板中 5 步分析法执行：

1. 从入口文件梳理启动流程，列出全局初始化和请求链路组件
2. 从通用目录提取响应格式、异常处理、通用约定
3. 理解认证授权机制
4. 理解日志/链路追踪/可观测性
5. 理解 API 文档生成方式

**归纳**：将零散发现按"调用方 / 开发者 / 运维"三分组，每组内容超过 2-3 句话的独立成子章节。不存在的直接跳过。

### 阶段 3：逐章填充

按模板顺序，每章先读模板中的 `<!-- 注释引导 -->`，再根据阶段 1-2 的信息填充。顺序：

技术栈 → 项目结构 → 快速开始 → 环境变量 → 常用脚本 → **系统架构**（阶段 2 落地）→ 业务模块 → 数据模型 → 测试 → 部署 → 条件章节（开发规范 / 变更流程，按需）

每章的判断规则：
- 模板中为 `{{PLACEHOLDER}}` 的 → 用实际值替换
- 模板注释标明"条件"的 → 项目有则写，无则整节删除
- 信息不足的 → 占位标注 `<!-- TODO: 待补充 -->`（仅限非核心章）

### 阶段 4：收尾

1. 对比模板章节清单，确认无遗漏
2. **回到文件顶部补全目录锚点**（先写正文再补目录）
3. `read_file` 通读验证：无残存 `{{...}}`、锚点正确、代码块标记正确
4. 运行 `npm run lint` 或等效检查（如果有）

## 原则

- **对调用方有用的优先**：响应格式、错误码、认证方式放前面，写详细
- **安装命令可执行**：给出的 shell 命令必须可直接复制粘贴运行
- **不做无中生有**：模板引导的 5 步分析出来 3 个就写 3 个，不强求"完整"
- **不写后端实现细节**：注释和说明面向 API 调用方和新加入的开发者
