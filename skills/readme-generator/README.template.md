## 系统架构

<!-- 
  ═══════════════════════════════════════════════════════════════
  ⚠️ 本章节是 README 的核心。不要直接套用任何预设的子章节标题——
  先阅读项目代码，归纳出本项目真正存在的架构特征，再决定写哪些子章节。
  ═══════════════════════════════════════════════════════════════

  ── 第一步：分析（框架无关） ──

  1. 找到程序的入口文件，理解启动流程：
     - 启动时初始化了哪些全局能力？（配置加载、数据库连接、消息队列、定时任务…）
     - 注册了哪些"全量请求都经过"的组件？（无论框架管它叫什么——中间件、拦截器、过滤器、钩子…）
     - 这些组件做了什么？按请求链路顺序把它们串起来。

  2. 找到项目中承载"通用能力"的目录/包（名称可能是 common、shared、core、lib、infra 等）：
     - 有没有统一的响应格式？成功/失败分别长什么样？
     - 有没有自定义的错误码/异常类？
     - 有没有通用的分页、排序、ID 生成等约定？

  3. 理解认证与授权：
     - 调用方如何证明身份？（header / cookie / token / mTLS / …）
     - 身份信息如何在代码中获取？（参数装饰器、上下文对象、thread-local…）
     - 有没有角色/权限模型？

  4. 理解可观测性：
     - 日志怎么打？有无统一的 logger 封装或使用约定？
     - 有没有链路追踪（traceId / spanId）？传播机制是什么？
     - 有没有请求访问日志？记录了哪些字段？

  5. 理解 API 文档（如果有）：
     - 用什么工具生成（Swagger / OpenAPI / GraphQL schema / …）？
     - 在什么环境可访问（如仅开发环境）？
     - 注释/类型如何映射到文档字段？

  ── 第二步：归纳为子章节 ──

  将上述分析得到的零散发现，按"谁需要知道"分组：
  - 对 API 调用方（前端/外部服务）有用的 → 响应格式、错误码、认证方式
  - 对开发者有用的 → 请求链路、日志约定、校验规则、文档生成
  - 对运维有用的 → 配置加载、健康检查、优雅关闭

  每个分组若内容超过 2-3 句话，就独立为一个子章节。
  如果某个分组在这个项目里不存在（如没有自定义错误码），直接跳过。

  ── 第三步：输出 ──

  - 子章节数量不做预设——分析出来 3 个就写 3 个，分析出来 8 个就写 8 个。
  - 如果子章节超过 7 个，考虑把次要的（如 CORS 配置、请求体大小限制）合并为简短段落。
  - 每个子章节：先用一句话给出结论，再给出关键细节（表格 > 列表 > 段落）。
  - 请求链路优先用 ASCII 流程图（兼容性最好），除非项目已使用 mermaid 等渲染器。
  - 对 API 调用方有直接影响的约定（响应格式、错误码、认证头）优先写、着重写。
